## Headline

x402watch's wash-detection algorithm is rebuilt around a 4-layer pipeline that separates seller signals from buyer signals. This release fixes a class of false positives where a single buyer touching multiple sellers' launch cohorts was globally mislabelled, inflating `suspected_wash %` on unrelated services. The release also ships an in-product dispute submission flow with rate limiting and auto-recompute.

No breaking changes to the public API contract. The schema and label taxonomy are versioned and documented at <https://x402.printmoneylab.com/docs/methodology>.

## Why we shipped this

Pre-v2.0, the labelling pipeline computed a single global label per buyer from a mix of seller-shaped and buyer-shaped signals. The most damaging signal — "launch-window concentration" — was fundamentally a *seller* property (was this seller's first-week traffic concentrated in a few wallets?) but was being attached to the *buyer* as `self_test` and then projected onto every service that buyer touched.

Concrete failure: buyer `0x15c3…ebc2b` (151 distinct sellers, 1,505 tx over weeks, multiple amount tiers — clearly an AI agent) was globally labelled `self_test`. KR Crypto kr-prices (`/services/14391`) — which they used legitimately — was reporting **96.4% suspected_wash**. Post-v2.0, the same service reads **100% real volume** with the buyer correctly labelled `ai_agent`.

## What's in v2.0

### Algorithm (Oracle backend)

- **New module** `indexer/seller_flags.py` (~332 lines) — per-seller flag computation. Priority chain: `owner_seller` → `confirmed_wash_farm` → `suspicious_launch` → `normal`.
- **New module** `indexer/pair_labels.py` (387 lines) — per-(buyer, seller) labels. Combines seller flag + buyer global features + multi-signal scoring with three global guards.
- **New module** `indexer/derive_global.py` — tx-weighted majority derivation of the global buyer label + a re-written service rollup that uses pair labels (so the same buyer can be `ai_agent` to service A and `developer` to service B).
- **New label** `owner_test` (9th label). Whitelisted operator wallets in `data/owner_wallets.json`. Excluded from both numerator and denominator of service rollups (neither demand nor fraud).
- **Atomic switch** with `*_pre_layer4_backup` tables on Postgres + `scripts/rollback_layer4.sh` for <2 minute recovery.
- `indexer/labeller.py` `run()` body replaced — old single-pass logic preserved as helpers for emergency rollback.

### Dispute submission (Step 6)

- **New tables** `label_disputes` + `recompute_queue` on Oracle Postgres. DB-layer unique index enforces "one dispute per reporter per buyer per day".
- **New FastAPI endpoints** on `api.x402.printmoneylab.com`:
  - `POST /api/v1/internal/disputes` (bearer auth, called by the CF Pages Edge proxy)
  - `GET /api/v1/internal/disputes/list` (admin)
  - `GET /api/v1/disputes/buyer/{address}` (public counts)
- **Edge proxy on Cloudflare Pages**: `POST /api/disputes` with four KV rate-limit scopes (10 / IP / hour, 1 / IP / buyer / 24h, 1 / IP / pair / 24h, 50+ / hour → 24h ban). Source: `src/app/api/disputes/`.
- **Auto-recompute**: at ≥ 5 independent pending disputes per buyer, the buyer is added to `recompute_queue`. The next daily labeller run drains the queue, compares the new label vs the dispute's snapshot, and closes each dispute as `resolved` (label changed) or `reviewed` (no change).
- **Telegram alert** on every new dispute, including pending count for the buyer and a 200-char reason excerpt.

### Dashboard (Cloudflare Pages frontend)

- **New component** `src/components/common/LabelBadge.tsx` — single source of truth for label rendering with confidence-band UX:
  - `≥ 0.85` → bold literal label
  - `0.70 – 0.84` → "likely …" + ⚠ link to `/docs/methodology#confidence-bands`
  - `< 0.70` → `unlabeled` (italic, muted, no-decision)
  - `owner_test` renders as "operator self-test"; bypasses the soft band.
- **New component** `src/components/common/ReportDialog.tsx` — native `<dialog>` modal, zero dep, accessible. Live char count (32-1000), per-error-code copy, confirmation state with `dispute_id`, GitHub Issue link kept as a secondary footer channel.
- **TopBuyersTable redesign** — splits external buyers from a collapsible "Operator self-tests" section so `owner_test` rows are visible but excluded from the public rollup.
- **Methodology page rewrite** — full v2.0 docs with Quick Reference, 4-layer architecture, confidence bands, known false-positive patterns, dispute process, Phase 2 roadmap. Legacy v1.x signals preserved in Appendix A.

## Migrations

Required for self-hosters:

1. Apply `oracle-patches/step6-disputes/disputes_schema.sql` to Postgres.
2. Place `oracle-patches/step6-disputes/disputes_api.py` at `app/disputes_api.py` and register the router from your FastAPI entrypoint.
3. Run `oracle-patches/step6-disputes/labeller_queue_patch.py` to splice `process_recompute_queue()` into `indexer/labeller.py`.
4. Set `X402_INTERNAL_TOKEN` on both Oracle `.env` and the Cloudflare Pages environment.
5. Re-deploy the Pages project (env-only redeploy).

Full deploy procedure with copy-paste commands + rollback path: `oracle-patches/step6-disputes/DEPLOY.md`.

## Breaking changes

None. The public REST contracts (`/api/v1/services/{id}`, `/api/v1/categories`) preserve their existing fields. The label taxonomy grew from 8 → 9 (added `owner_test`); existing clients that hard-code the 8 labels will see an unknown label string but no schema break.

## Validation

- KR Crypto kr-prices (`/services/14391`): pre-v2.0 96.4% wash → post-v2.0 100% real.
- Aubrai chat (`/services/14239`): pre-v2.0 ~95% wash → post-v2.0 99.98% wash (true positive preserved).
- 0x15C3 global label: pre-v2.0 `self_test` → post-v2.0 `ai_agent` with `reason = "derived_from_pairs:ai_agent(98%),organic_user(2%)"`.
- Owner test buyers correctly isolated: 7 EVM + 1 SVM whitelisted wallets get `owner_test` globally, excluded from service rollups.

## Acknowledgments

The KR Crypto kr-prices false positive was identified and triaged by Moa during a routine ecosystem review. The decision to do a full 4-layer redesign rather than a surgical fix is the core engineering call this release reflects — "단편 fix는 또 다른 false positive 만들 위험".
