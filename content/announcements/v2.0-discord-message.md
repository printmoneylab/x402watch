Hey x402 operators 👋

Quick heads-up: we just shipped v2.0 of the wash-filter algorithm on x402watch (https://x402.printmoneylab.com). The change is fully internal — no breaking changes to the public API or your endpoints — but it directly improves how your service's `real_volume_pct` is computed.

**What changed**

The previous algorithm had a bug where a per-seller signal ("launch-window concentration") was being applied to buyers *globally*. So an LLM agent that happened to touch a few launch cohorts got globally labelled `self_test` — which then inflated `suspected_wash %` on every *other* service that buyer used. Some services were reading 90%+ wash that were actually 100% real traffic.

v2.0 restructures this into a 4-layer pipeline:

1. `seller_flags` — per-seller signals (`normal` / `suspicious_launch` / `confirmed_wash_farm` / `owner_seller`)
2. `pair_labels` — per (buyer, seller) labels combining seller flag + buyer features
3. **Global guards** — e.g. a buyer with ≥ 10 distinct sellers can't be `self_test` on any pair
4. `derive_global` — global buyer label = tx-weighted majority across pairs

We also added a 9th label `owner_test` for operator self-test traffic, which is now *excluded* from both real-volume and wash denominators (it's neither demand nor fraud, so neither percentage should be compressed by it).

**What you can do**

- Check your service's revised stats: `https://x402.printmoneylab.com/services/<your-id>`
- If a buyer is labelled `self_test` or `suspected_wash` and you believe it's wrong, click the flag icon next to the label. The dispute dialog submits to a rate-limited endpoint (`/api/disputes`); 5+ independent reports auto-queue a recompute on the next daily run.
- If you operate test wallets that should be permanently excluded, send me a DM (or open a GitHub Issue) — we add verified wallets to `data/owner_wallets.json` and they get the `owner_test` label.

**Methodology + dispute audit log**

Full v2.0 documentation, 9-label taxonomy, confidence bands, known false-positive patterns, and the dispute process are all open:

Feedback welcome — especially from operators who saw their `wash_pct` drop significantly after this update. We'd like to know which corrections felt correct and which still don't.
